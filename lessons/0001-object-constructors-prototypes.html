Object Constructors & Prototypes

How JavaScript builds "classes" out of plain functions and live objects — taught through the Java you already know.

Total: ~45 min

Why This Matters Prototypes and this are the two places where your Java intuition is actively wrong, not just incomplete — and they're top-tier JS interview topics. Everything React and Node does with objects sits on this machinery. Master it here and the upcoming Classes lesson becomes pure syntax.

1. The Big Picture: Java Bundles, JS Splits ⏱ 4 min

In Java, a class is one sealed unit: fields, constructors, instance methods, statics — everything a type is, declared together, frozen at compile time. JavaScript takes that bundle and splits it into two live pieces:

Java class partWhere it lives in JS
Constructor + field declarationsA plain constructor function — function Book(...)
Instance methods (shared, one copy)The prototype object — Book.prototype
The hidden "which class am I?" pointerEach instance's hidden [[Prototype]] link
java.lang.Object at the topObject.prototype at the end of the chain

Nothing here is conceptually new to you — Java does all of this too. The difference: Java hides the machinery; JS hands you the parts. The method table is a real object you can inspect and edit at runtime. That visibility is the whole lesson.

Key Insight You have used prototypes for years without knowing it. Every time Java shared one method implementation across 1,000 instances, that was a prototype by another name — just sealed away from you.

2. Constructor Functions & new ⏱ 8 min

Terms in this section
constructor function — a plain function used with new; capitalized by convention only
new — the operator that turns a function call into instance creation
new.target — meta-property: set when called with new, undefined otherwise
this — inside a new-call: the freshly created object

There is no class required. A constructor is just a function that assigns onto this:

function Player(name, marker) {
  this.name = name;
  this.marker = marker;
}

const player1 = new Player("steve", "X");
player1.name; // "steve"
The Java equivalent is the constructor body alone — no class wrapper exists.

What new actually does — four steps, memorize them:

new Player("steve", "X"), step by step

1Create a brand-new empty object {}
2Link its hidden [[Prototype]] to Player.prototype (the "stamping" — more in §3)
3Run the function body with this = that new object (so this.name = name writes onto it)
4Return the object automatically — no return statement needed
player1 → { name: "steve", marker: "X" } + hidden link to Player.prototype
Trap — impossible in Java, routine here Calling Player("steve", "X") without new doesn't error by default. It runs as a plain function, this is undefined (strict mode) or the global object, and your properties leak into nowhere. Java's compiler makes this mistake unrepresentable; JS needs a guard: 
function Player(name, marker) {
  if (!new.target) {
    throw Error("You must use the 'new' operator to call the constructor");
  }
  this.name = name;
  this.marker = marker;
}

Granular Java diffs worth 10 seconds each: there is no overloading (one function per name — the last definition silently wins, and any function accepts any number of args); an explicit return <object> inside a constructor hijacks the result while returning a primitive is silently ignored; and const player1 is Java's final reference — binding locked, object still fully mutable.

Practice: Song constructor

domain: music — you play guitar, use it

Q: Write a guarded constructor Song(title, artist, bpm) and instantiate one. Then, without running it: what does mySong.genre evaluate to — error or something else, and why is that different from Java?

function Song(title, artist, bpm) {
  if (!new.target) throw Error("use new");
  this.title = title;
  this.artist = artist;
  this.bpm = bpm;
}
const mySong = new Song("Neon", "John Mayer", 122);

mySong.genre → undefined. No compile error, no exception — JS returns undefined for any missing property. Java would refuse to compile. This is the #1 silent-bug class for Java devs: typos fail three functions later, not at the read.

3. The Two Things Named "Prototype" ⏱ 9 min

This is the confusion point for almost everyone, so here is the honest framing: JavaScript uses one word for two different slots.

The two slots
Fn.prototype — a visible property that only FUNCTIONS have
[[Prototype]] — a hidden link that every OBJECT has (read: Object.getPrototypeOf(obj))

The factory analogy. The constructor is a factory, instances are products, and the prototype object is a service center where shared capability lives:

Fn.prototype — the card on the factory wall The factory keeps an address card pinned up: "stamp THIS service-center address on every product we ship." It exists from the moment the function is defined — before any instance. It does nothing by itself.
[[Prototype]] — the stamp on the product Every product rolls off the line with the address stamped on its underside: "if I can't do something, I go here." Stamped once, at that product's creation moment, by new.

Same address, two locations. That's why this is always true:

Object.getPrototypeOf(player1) === Player.prototype; // true — the stamp equals the card it was copied from

Java mapping: [[Prototype]] ≈ the hidden class-pointer in every Java object's header. Fn.prototype has no Java equivalent at all — Java never gives you a handle on "what future instances will point to." JS hands you the card.

And methods? Data goes in the constructor; behavior goes on the card:

function Player(name, marker) {
  this.name = name;      // data: per-instance, in the constructor
  this.marker = marker;
}

Player.prototype.sayName = function() {   // behavior: ONE shared copy
  return this.name;
};
1,000 players, one sayName. Exactly what Java's method area does for you silently.
Trap — methods inside the constructor this.sayName = function() {...} inside the constructor works, but stamps a private copy of the function into every instance — like every Java object carrying its own duplicate of every method. Legal, occasionally deliberate (closures, later lesson), but the default is: data on the instance, methods on the prototype.
Key Insight .prototype is not a method — it's a data property. And it can sit on a function because functions are objects in JS (callable objects that carry properties). One-way street: objects are NOT functions. Think Function extends Object.

4. Live Lookup — Mutation vs Reassignment ⏱ 9 min

Property lookup runs fresh at every access: check the object's own properties; miss → follow the stamp; check the card as it is right now. The card is live. This has a beautiful consequence and a nasty trap, and they look almost identical:

Dog.prototype.bark = function() {...}  // MUTATE the card  → ALL instances, past + future ✅
Dog.prototype = { bark() {...} }       // REPLACE the card → FUTURE instances only ⚠️

You already know this distinction from Java — it's reference semantics:

list.add(x);              // mutate: everyone holding this list sees x
list = new ArrayList<>(); // reassign: old holders keep the OLD list, unaffected

The rex / pup experiment — run this in your console

1function Dog(name) { this.name = name; } — card created, empty-ish
2const rex = new Dog("Rex") — rex stamped with the ORIGINAL card's address
3Dog.prototype = { bark() { return "woof"; } } — factory swaps in a NEW card. Nothing re-stamps rex
4const pup = new Dog("Pup") — pup stamped with the NEW card
pup.bark() → "woof" ✅   rex.bark() → TypeError ❌ — rex's lookup IS live, but live against the original object, which never got bark
The rule, one sentence A prototype method must exist at call time, and the dangerous operation is reassignment (Fn.prototype = {...}), never mutation (Fn.prototype.m = ...). The stamp is the address of the building, not a photo of its contents on stamping day — renovate freely; demolish-and-redirect splits history.

Practice: Predict the output

domain: pokemon

Q: function Pokemon(name) { this.name = name; } → const pika = new Pokemon("Pikachu") → Pokemon.prototype.attack = function() { return this.name + " used Thunderbolt!"; } → what does pika.attack() return, and why does this NOT contradict the rex case?

"Pikachu used Thunderbolt!" — works fine.

This is MUTATION (writing onto the existing card via dot), so pika's live lookup finds attack at call time even though pika was created first. rex broke because his card was REPLACED (=), orphaning him on the old object. Dot-write = renovation; equals-write on .prototype = demolition.

5. The Prototype Chain ⏱ 6 min

The card is an object, so it has a stamp too. Miss on the card → follow ITS stamp. Repeat until the chain ends:

player1 → Player.prototype → Object.prototype → null
// own props    sayName            toString, valueOf,     end of
//  name, marker                    hasOwnProperty         the line

Object.prototype is JS's java.lang.Object: the shared ancestor supplying defaults. That's where player1.valueOf() comes from when you never defined it. Prove where things live:

player1.hasOwnProperty("name");     // true  — own furniture
player1.hasOwnProperty("sayName");  // false — found up the chain (Player.prototype)
player1.hasOwnProperty("valueOf");  // false — two hops up (Object.prototype)
Object.getPrototypeOf(Object.prototype); // null — chain over; misses become undefined
java.lang.Object gives youObject.prototype gives you
toString()toString() — default "[object Object]", override the same way
equals() / hashCode()Nothing. No equality protocol exists; === is reference-compare, not overridable
(no equivalent)valueOf() — object→primitive coercion hook (numeric contexts)
(can't ask this)hasOwnProperty() — "mine, or inherited?" Only meaningful with visible chains
Trap — shadowing An own property with the same name as a prototype property wins lookup for that one object only (own props are checked first). Like Java field hiding, but per-object, at runtime, silently. player1.sayName = ... masks the shared method for player1 alone.

6. Prototypal Inheritance ⏱ 6 min

Java: class Player extends Person — a compile-time declaration between types. JS: physically link the two cards at runtime, so a miss on Player's card forwards to Person's:

function Person(name) { this.name = name; }
Person.prototype.sayName = function() { return `Hello, I'm ${this.name}!`; };

function Player(name, marker) {
  this.name = name;
  this.marker = marker;
}
Player.prototype.getMarker = function() { return `My marker is "${this.marker}"`; };

Object.setPrototypeOf(Player.prototype, Person.prototype); // link card → card

const p = new Player("steve", "X");
p.getMarker(); // found on Player.prototype (1 hop)
p.sayName();   // miss, miss, found on Person.prototype (2 hops)

Chain now: p → Player.prototype → Person.prototype → Object.prototype → null. Rules: set the link before creating instances (rewiring after is a perf hit), and single inheritance only — one stamp per object.

Trap — the HAHAHA bug (aliasing ≠ inheritance) Player.prototype = Person.prototype doesn't inherit — it makes two factories share ONE physical card. When a third "class" (Enemy) does the same and then edits sayName, it defaces the shared card: every Player now screams HAHAHA too. Java analog: two classes literally sharing one method table. Link cards (setPrototypeOf), never merge them (=).

7. this — Five Rules, Not One ⏱ 7 min

Java has exactly one rule: this = the enclosing instance, fixed forever at compile time. JS re-decides this fresh at every call, based on HOW the function is called. It's a hat the function borrows from whoever's wearing it at the call site:

How the function is calledWhat this is
obj.method()obj — whatever is left of the dot
plainFunction()undefined (strict) / global object (sloppy)
new Fn()the freshly created object
fn.call(x) / apply / bindx — you choose explicitly
arrow function =>no own this — inherits from where it was written

Priority when rules collide: new > bind/call > dot > plain.

Trap — the detached method (impossible in Java) 
rps.playRound("rock");            // this = rps. Fine, feels like Java
setTimeout(rps.playRound, 1000);  // method DETACHED from the object —
                                  // when the timer fires, nothing is left
                                  // of the dot → this is NOT rps ❌
Java cannot even express "rip the method off the object" — in JS it's routine (callbacks everywhere) and it's a whole genre of bugs. The fix arrives in later lessons (bind, arrows-as-callbacks).
Trap — arrows as methods Arrow functions don't borrow the hat — they keep the this of the scope where they were written. An arrow as an object-literal or prototype method therefore never sees the instance. Rule for now: shorthand/function expressions for methods, never arrows.

Practice: closure vs this — the interview classic

domain: the Book exercise you just did

Q: A Book constructor defines info() INSIDE the constructor reading the bare parameters (`${title}, ${pages} pages`) instead of this.title/this.pages. It works. Later someone does book.pages = 300 and calls book.info(). What prints, and why?

It still prints the OLD page count (e.g. 295). The inner function read the constructor's PARAMETERS via closure — variables captured at construction — not the object's properties. book.pages = 300 updated the object; the closure variables never change. Reading through this.pages follows the live object and prints 300. Two storage locations, one stale. Moving info() to the prototype forces the fix: prototype methods have no access to constructor parameters, so everything must ride in on `this`.

9. Quiz ⏱ 5 min

1. Which of these is NOT one of the four things new does?

  • Copies the prototype's methods into the new object
  • Creates a fresh empty object
  • Links the object's [[Prototype]] to Fn.prototype
  • Runs the function with this bound to the new object
Nothing is ever copied — that's the whole point. Instances hold a LINK (the stamp), and lookup walks to the shared card at call time. If methods were copied, mutating the prototype later couldn't reach existing instances (but it does).

2. const rex = new Dog("Rex"), then Dog.prototype = { bark() { return "woof"; } }, then const pup = new Dog("Pup"). What happens?

  • Both bark — the prototype is live
  • pup barks; rex throws TypeError
  • rex barks; pup throws TypeError
  • Neither barks until instances are re-created
Reassignment REPLACES the card. rex keeps his original stamp (old object, no bark) — stamps are never updated. pup, created after the swap, points at the new card. Mutation (Dog.prototype.bark = ...) would have reached both.

3. Dog.prototype vs Object.getPrototypeOf(rex) — the precise relationship?

  • Same thing, two syntaxes
  • The first is the function's template for FUTURE instances; the second reads an existing object's live link — equal only until the card is replaced
  • The first is the parent class, the second the child
  • getPrototypeOf returns a copy, .prototype the original
Card on the factory wall vs stamp on the product. They point at the same object right after new, but they are different slots — reassign Dog.prototype and existing stamps stop matching it (the rex/pup split).

4. setTimeout(rps.playRound, 1000) — inside playRound when the timer fires, this is…

  • rps, because the method belongs to it
  • setTimeout
  • undefined / the global object — the method was detached from rps
  • A copy of rps frozen at schedule time
this is decided at CALL time by what's left of the dot. Passing rps.playRound hands over the bare function; when the timer calls it, nothing is left of the dot. Java's this is fixed at compile time — this bug class doesn't exist there.

5. Why do methods belong on the prototype instead of inside the constructor?

  • Constructor methods can't use this
  • One shared copy vs a private duplicate stamped into every instance
  • Prototype methods run faster per call
  • It's only a style convention
1,000 instances with constructor-defined methods = 1,000 function copies (every apartment gets its own washing machine). On the prototype: one shared laundry room, found via the chain. It's exactly what Java's method area does for you invisibly.

6. JS's closest analog to java.lang.Object?

  • The Object function itself
  • Object.prototype — end of (nearly) every chain, supplying toString/valueOf/hasOwnProperty
  • globalThis
  • There is none — JS objects share no ancestor
Object.prototype is the master binder at the back of the room. Note the split: instance-inherited defaults live there; static-feeling utilities (Object.keys, Object.assign, Object.getPrototypeOf) live on the Object FUNCTION and are not inherited — that's why it's never player1.getPrototypeOf().

7. Player.prototype = Person.prototype — what did you just do?

  • Made Player inherit from Person
  • Aliased two "classes" onto ONE shared card — editing one now defaces the other (the HAHAHA bug)
  • Copied Person's methods into Player.prototype
  • Nothing — it throws
That's aliasing, not inheritance — one physical object, two names. Inheritance links two separate cards so misses forward: Object.setPrototypeOf(Player.prototype, Person.prototype), done before creating instances.
